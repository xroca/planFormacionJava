1.   Introducción.

 

Uno de los aspectos clave en cualquier aplicación seria es el acceso a bases de datos, donde se almacena la información crítica para determinados procesos de negocios. Es por este motivo por el que cualquier entorno que pretenda calar entre los desarrolladores ha de proveer los mecanismos adecuados para hacerlo posible.

 

Java, como no podía ser menos, proporciona accesibilidad a bases de datos relacionales desde la versión 1.1. Para ello, emplea un conjunto de clases que forman una API (Application Program Interface) denominada JDBC (Java Database Connectivity).

 

Este API provee toda la funcionalidad requerida para realizar las operaciones más habituales sobre las bases de datos.  Además, es independiente de cualquier base de datos dentro de ciertos límites que comentaremos más adelante, con lo que no es necesario ningún cambio en la programación de la aplicación si en cualquier momento se pretende cambiar a otra base de datos que sirva como soporte.

 

En este capítulo, estudiaremos cómo se emplea, comenzando por la arquitectura básica que provee el estándar JDBC.  Todas las clases e interfaces que vamos a manejar se encuentran dentro del paquete java.sql.

 

2.   Arquitectura.

 

JDBC se inscribe por completo en la arquitectura cliente/servidor multicapa, y permite realizar cuatro operaciones básicas:

 

q       Establecer una conexión con una base de datos.

q       Enviar órdenes SQL a la base de datos.

q       Obtener los resultados de tales sentencias.

q       Desconectar de la base de datos.

 

Las cuales quedan cubiertas por sendas clases y/o interfaces que modelan cada uno de estos puntos, colocadas en el nivel más alto de abstracción de la arquitectura JDBC.

 

Para realizar la programación de bases de datos por medio de JDBC es necesario emplear SQL bien como lenguaje de definición de datos (DDL), bien como lenguaje de manipulación de datos (DML).

La arquitectura JDBC está diseñada para maximizar la independencia de la base de datos física y la facilidad de mantenimiento.

 

2.1.   Componentes.

 

La arquitectura JDBC se escinde en tres niveles distintos.  Estos son:

q       Llamadas al API JDBC: con el que el programador realiza los accesos a la base de datos. Es el de más alto nivel de los tres.

q       El gestor de drivers o driver manager. Se encarga de almacenar todos los drivers empleados por una aplicación, así como de proporcionar un canal de conexión con la base de datos física. Es el responsable en gran medida de la independencia con respecto a las bases de datos. El driver manager se basa en las URLs. de las bases de datos para asignar la gestión a los distintos drivers.

q       Los drivers. En este nivel, el más bajo de todos, es donde se encuentran los verdaderos objetos que conocen el protocolo de la base de datos con la que pueden conectar. Para realizar una conexión a una base de datos, debemos disponer de un driver que pueda contactar y entenderse con el gestor de bases de datos empleado. Todo driver compatible con JDBC debe implementar al menos la funcionalidad definida en el punto de entrada ANSI SQL-2. Un driver JDBC no es más que una clase Java que implementa la interfaz Driver.

2.2.   Gestor de drivers.

 

El gestor de drivers o driver manager es el encargado de proporcionarnos una conexión con una base de datos física a través del driver concreto que sabe entendérselas con ella.

 

Para realizar esto, el gestor de drivers lleva un registro interno en el que guarda una equivalencia directa entre tipos de URLs (que definen la base de datos a la que conectar, el protocolo y la máquina en la que se encuentra) y los drivers que las gestionan.

 

Será misión del programador registrar dentro del gestor de drivers los drivers adecuados para las bases de datos que emplea.  Esto se realiza con una simple llamada que se estudiará más adelante.

 

En caso de que en un futuro se deba cambiar la base de datos o si nuestra aplicación tiene la capacidad de acceder a distintas bases de datos controladas por gestores de bases de datos heterogéneos, siempre podemos facilitar más de un driver. Además, la carga dinámica de clases en Java posibilita el hecho de añadir nuevos gestores bajo demanda.

 

2.3.   Tipos de drivers.

 

El estándar JDBC define cuatro tipos de drivers distintos:

 

El puente JDBC-ODBC y los drivers ODBC.

Incluidos con toda distribución de Java a partir de la versión 1.1. El estándar ODBC (Open Database Connectivity) de Microsoft es de amplia difusión para el acceso a bases de datos desde prácticamente cualquier lenguaje bajo Windows.

Así, el puente JDBC-ODBC es un medio de aprovechar todo lo relacionado con ODBC para realizar programación de bases de datos desde Java.

Este puente traduce las llamadas a JDBC en llamadas a ODBC, aislando de nuevo al usuario de cualquier dificultad. Además del puente, es necesario que exista un driver ODBC para el gestor de bases de datos particular que vamos a emplear.

La ventaja que tiene este primer tipo de drivers es que aprovecha lo ya hecho para ODBC, y podemos acceder a cualquier base de datos compatible con ODBC como Access, SQL-Server, Oracle, Sybase, etc...

La gran desventaja que presenta este driver es que necesita código nativo en la máquina cliente en la que se ejecuta, con lo que eso implica en relación con aspectos de seguridad.

Driver de API-nativo, y parcialmente escrito en Java.

 Este tipo de drivers se salta el puente, convirtiendo directamente las llamadas al API JDBC en llamadas a los gestores de bases de datos particulares. Como gran inconveniente presentan la necesidad ejecutar código nativo en el cliente.

Driver JDBC de red totalmente escrito en Java.

Este tipo de drivers no presenta la desventaja de necesitar código nativo, pues están completamente escritos en Java. Lo que los hace peculiares es que no se conectan directamente con la base de datos, sino que existe una capa software intermedia (middleware) que es la que realmente lleva a cabo las conexiones.

Como ventaja se tiene la gran uniformidad de acceso a bases de datos que esto presenta, pues es el software intermedio el que se encarga de realizar los accesos. Como inconveniente, destacar que existe esa capa intermedia que puede relentizar las operaciones sobre la base de datos. En este tipo de drivers se encuentran englobados algunos productos como el Borland Gateway.

Driver JDBC de protocolo puro escrito totalmente en Java.

Al igual que los anteriores, se encuentran escritos enteramente en Java. Lo que los distingue es que en esta ocasión el acceso se hace directamente sobre el gestor de base de datos, normalmente a través de Sockets, en lugar de emplear middleware. Dentro de este apartado podemos englobar los drivers thin de Oracle, por ejemplo.

 

2.4.   URLs.

 

Como ya hemos comentado, la forma en que el gestor de drivers elige el adecuado es por medio del uso de una URL (Uniform Resource Locutor).

 

Cuando el programador abre una conexión con una base de datos, necesita indicarle al gestor de drivers la URL que permite identificarla. Como en cualquier URL, se especifica el protocolo, servidor y puerto a los que conectarse.

 

La estructura general de una URL JDBC es la siguiente:

 

<protocolo>:<subprotocolo>:<subnombre>

 

Cada parte se separa por dos puntos.

 

La primera parte especifica que el protocolo a emplear en la comunicación A día de hoy, el protocolo es siempre el mismo: jdbc.

 

El subprotocolo es el nombre del driver o el nombre del mecanismo por el cual un driver puede reconocer que la conexión va dirigida a una base de datos que él puede tratar. Por ejemplo, si estamos empleando el puente JDBC-ODBC, la URL toma la forma:

 

jdbc:odbc:dsn

 

Donde dsn (el subnombre) es la abreviatura de data source name o nombre de la fuente de datos, término este tomado de la especificación ODBC. Por el contrario, si por ejemplo estamos usando BorlandBroker, la URL tiene la forma:

 

jdbc:BorlandBroker://servidor:puerto/base-datos

 

Como se puede apreciar, las URLs sólo son comunes en su estructura en el protocolo. El resto depende del driver empleado. En los ejemplos anteriores, tanto dsn como servidor:puerto/base-datos, especifica un nombre por medio del cual localizar una base de datos, papel reservado siempre al subnombre

 

Como caso particular, mencionar que cuando se use el puente JDBC-ODBC, pueden pasarse parámetros adicionales al driver justo después del data source name. Estos parámetros e pasan por medio de un atributo y su valor y se encuentran separados por puntos y coma. Por ejemplo jdbc:odbc:misdatos;UID=yo;PWD=clave.

Pasaremos ahora a estudiar las distintas llamadas al API JDBC.

 

3.   Clases del API JDBC.

 

En esta sección vamos a estudiar las distintas clases/interfaces que hacen posible el control de bases de datos desde Java. Cubriremos las versiones 1.0 y 2.0 de JDBC.

 

Para ello, empezaremos viendo en primer lugar cómo obtener una conexión a una base de datos y aprenderemos a configurar los parámetros adicionales necesarios para el uso del puente JDBC-ODBC en una máquina basada en Windows. Proseguiremos viendo cómo enviar sentencias SQL a los gestores de bases de datos y cómo obtener los resultados. También nos encargaremos del tratamiento de errores en los programas que usan JDBC, estudiando las oportunas excepciones.

 

3.1.   Carga de drivers.

 

Para poder usar una conexión a una base de datos hemos de disponer del driver adecuado. Existen varias formas de indicar a Java que registre una clase como driver, pero la más empleada es la mostrada a continuación:

 

Class.forName (paquete_driver.nombre_clase_driver);

 

Esta sentencia debe ser invocada antes de intentar establecer cualquier conexión con el sistema gestor que controla el driver que se carga.

 

El resultado de esta sentencia es la carga de la clase que se especifica. Si el driver se ha programado adecuadamente, es decir, se ha seguido la especificación de Sun, ello mismo se autoregistra en el driver manager invocando al método DriverManager.registerDriver(), el cual lo incluye en la lista de drivers activos.

Resulta imprescindible, no obstante, estudiar en profundidad la documentación del driver suministrada por el fabricante para asegurarnos de cómo deberíamos cargarlo.

Si existe algún error al localizar la clase (por ejemplo esta no se encuentra en el CLASSPATH),  se lanza una excepción de la clase ClassNotFoundException.

Es posible cargar más de un driver en una aplicación. Si así lo hacemos, al solicitar una conexión el driver manager interroga a cada uno de ellos si puede devolverla. La conexión obtenida proviene del primer driver que responda sí.

Otra posible forma de cargar uno o varios drivers es dar valor a la propiedad jdbc.drivers.

Cuando el DriverManager se inicializa, la busca, y si existe intentará cargar todos los drivers enumerados por ella.

Por ejemplo:

java -Djdbc.drivers=uno.Driver:dos.sql.Driver programaJava

 

Si decidimos emplear el driver que Sun nos da por defecto en J2SE, la orden sería:

Class.forName(sun.jdbc.odbc.JdbcOdbcDriver);

 

3.2.   Conexiones.

 

Una vez hemos cargado el driver, ya podemos establecer una conexión. Las conexiones a las bases de datos quedan encapsuladas por medio de un objeto de la interfaz Connection.

 

Las conexiones no se crean por medio de un constructor, sino que han de obtenerse del gestor de drivers, pues como ya hemos visto es el que puede discernir qué driver ha de tratar nuestra conexión. Por tanto, el gestor de drivers actúa como una factoría de objetos.

 

Así, para obtener una conexión, hemos de usar un código como el siguiente:

 

Connection con = DriverManager.getConnection

(url, usuario, password);

 

Los tres parámetros que toma el método estático getConnection() de la clase DriverManager son Strings.  El primer parámetro representa la URL que indica dónde se encuentra la base de datos. Los dos siguientes parámetros definen el nombre del usuario y su password, respectivamente.

 

Por ejemplo, supongamos que queremos conectar a una base de datos ODBC denominada agenda, con el nombre de usuario administrador y clave admkey. El código a escribir sería:

 

Connection con = DriverManager.getConnection

(jdbc:odbc:agenda, administrador, admkey);

 

Una vez tenemos una conexión con la base de datos, podemos realizar todas las operaciones que detallaremos en las siguientes secciones.

 

En caso de que haya algún error al conectar con el gestor de la base de datos, se lanza una excepción del tipo SQLException. Estas excepciones tienen la peculiaridad de que pueden lanzarse más de una al mismo tiempo, en cuyo caso se lanza la cabeza de la lista que contiene todas las excepciones.

 

3.3.   Sentencias.

 

Ahora que ya disponemos de una conexión con la base de datos, ya podemos ejecutar las sentencias SQL que necesite nuestra aplicación particular.

 

La clase más sencilla para lograr este propósito es aquella que implementa la interfaz Statement. Al igual que las conexiones, las sentencias no se obtienen directamente creando objetos a partir de su constructor, sino que debemos obtenerlas de la conexión.

 

Para obtener un objeto que nos permita ejecutar una sentencia, podemos emplear el siguiente fragmento de código:

 

Statement stmt = con.createStatement();

 

Aquí con es una referencia válida a una sentencia.  Esta invocación puede lanzar una excepción del tipo SQLException.

 

Esto crea la sentencia.  Ahora podemos ejecutar la sentencia SQL que deseemos.  Para ello, dos de los métodos más empleados son:

 

stmt.executeQuery (SELECT );

stmt.executeUpdate(INSERT | UPDATE | DELETE | CREATE | );

 

El primer método se utiliza para realizar consultas a la base de datos, es decir, sentencias del tipo SELECT.  Este método devuelve una referencia a un objeto del tipo ResultSet, que contiene los resultados de tal consulta.  Esta clase se estudia más adelante.

 

Por el contrario, el segundo método se emplea con sentencias SQL que requieren hacer alguna modificación en la base de datos: creación de tablas, borrado, inserciones, actualizaciones, etc. Devuelve el número de elementos afectados por la acción ejecutada (por ejemplo, en una actualización el número de filas modificadas).

 

3.4.   Sentencias preparadas.

 

Si bien las sentencias encapsuladas por medio de Statement son válidas para la gran mayoría de las ocasiones, adolecen de una notable falta de rendimiento cuando se utilizan de forma continuada.

 

La causa es que cada vez que se ejecuta una sentencia por medio de executeQuery() o executeUpdate(), ha de pasar por una compilación previa a su ejecución, gastando algo más tiempo.

 

Por otra parte, las sentencias no son reutilizables. Supongamos que queremos realizar varias inserciones en la base de datos. La sentencia SQL es la misma; tan sólo cambian los valores de las columnas que queremos insertar. Sería deseable poder parametrizar la sentencia y reutilizarla.

 

        Ambos inconvenientes de una sentencia tipo Statement, relativa ineficiencia, y escasa capacidad de reutilización, pueden solventarse por medio de las sentencias preparadas.

 

Una sentencia preparada se compila una sóla vez y se puede ejecutar muchas veces con valores parametrizados La interfaz PreparedStatement deriva de Statement, con lo que toda la funcionalidad de las sentencias anteriormente descritas se mantiene.

 

        Supongamos por ejemplo que estamos construyendo una aplicación que gestiona información acerca de los nuevos clientes de una empresa en expansión. Esta información está compuesta del nombre, apellidos y teléfono del cliente. Además estos datos se almacenan en una tabla llamada datos. Dado que los datos de las personas van a ser siempre los mismos, podemos escribir lo siguiente:

 

PreparedStatement pstmt = con.prepareStatement

(insert into datos values (?,?,?));

 

Esto construye una sentencia preparada (se precompila antes de ser usada por primera vez), donde los ? simbolizan los parámetros a sustituir en cada uso de la sentencia preparada.

 

Así, en alguna parte del programa podría aparecer un código como el siguiente:

 

String nombre = << obtener el nombre de alguna forma >>

String apellidos = << obtener los apellidos de alguna forma >>

long telefono = << obtener el teléfono de alguna forma >>

// Sustituimos los parámetros.

pstmt.setString (1, nombre);

pstmt.setString (2, nombre);

pstmt.setLong (3, telefono);

 

Existe un método setXXX() para cada tipo básico (incluido Object).  Estos métodos toman dos parámetros: el primero es la posición del parámetro a sustituir (comenzando desde 1) y el segundo el valor a fijar.  Además estos métodos llevan a cabo ciertas traducciones para convertir los tipos de datos de Java a los empleados por SQL.

 

Esto incrementa de forma notable la velocidad de ejecución de las sentencias. Para ejecutar las sentencias ahora se emplea los métodos executeQuery() y executeUpdate(), esta vez sin argumentos.

 

Hay que tener cuidado si queremos fijar un parámetro a null, en cuyo caso debemos usar el método setNull(). Este método toma dos parámetros: el lugar del argumento a sustituir y el tipo del argumento. Los tipos de los argumentos están definidos por sendas constantes en la clase Types. Por ejemplo, para fijar un campo de tipo integer a null, emplearíamos el siguiente código:

 

pstmt.setNull (i, Types.INTEGER);

En una sentencia preparada sólo podemos parametrizar los nombres de las columnas.

 

3.5.   Procedimientos almacenados.

 

Existen determinados gestores de bases de datos que permiten que ciertos procedimientos escritos en algún lenguaje derivado de SQL se compilen y almacenen en él, de manera que puedan ser invocadas como subrutinas desde cualquier lenguaje de programación.

 

La forma de emplear estos procedimientos en JDBC es por medio de la interfaz CallableStatement.    Debe obtenerse como los anteriores por medio de una conexión activa:

 

CallableStatement cstmt = con.prepareCall ({call ProcAlmacenado (?,?,?, , ?)});

 

La sentencia anterior dejaría disponible una sentencia para invocar un procedimiento almacenado en el gestor de base de datos.  Como CallableStatement deriva de PreparedStatement, todos los métodos de la segunda interfaz están disponibles en la primera.

 

En el ejemplo anterior nada impide que algunos de los parámetros ya tengan valores fijos que no necesiten ser actualizados.

 

Sin embargo, un procedimiento también puede retornar parámetros. En este caso, nuestro programa podría tomar el valor de retorno. Para esto se usaría un código como el siguiente:

 

CallableStatement cstmt = con.prepareCall ({? = call ProcAlmacenado (?,?,?, , ?)});

 

En este caso, los parámetros se empiezan a numerar desde el parámetros de retorno-

 

Para recoger tanto el valor de retorno de una función o el valor de un parámetro de entrada/salida o sólo salida que se pasa a un procedimiento almacenado se emplea el método registerOutParameter().  Este método toma dos parámetros: el primero es el lugar del argumento de salida y el segundo el tipo SQL definido en la clase Types.

 

Una vez se ha ejecutado la llamada por medio del método execute(), pueden obtenerse estos valores por medio de los métodos getXXX(), donde XXX es un tipo de datos al que se puede convertir el tipo de datos SQL.  Por ejemplo, supongamos el procedimiento almacenado llamado CrearPersona que toma dos parámetros: el DNI y el nombre.  Además, devuelve un número único que será el código de la persona.  Para obtener este número, podríamos emplear el código siguiente:

 

CallableStatement cstmt = con.prepareCall (? = call CrearPersona (?,?));

cstmt.registerOutParameter (1, Types.INTEGER);

cstmt.setInt (2, 13456789);

cstmt.setString (3, Bart);

cstmt.execute();

int n = cstmt.getInt(1);

 

Es necesario mencionar que puede preguntarse si un parámetro de salida es null invocando el método wasNull() después de una llamada a un método getXXX().

 

Además, como nota final, deberíamos obtener siempre los parámetros en órdenes crecientes de posición, pues no todos los drivers permiten acceder a cualquier parámetro en cualquier momento.  Para restringir aún más, la llamada obteniendo el parámetro debería hacerse una sola vez.

 

3.6.   Conjuntos de resultados.

 

Una vez vistos los distintos tipos de sentencia que pueden usarse en nuestros programas, pasamos a ver cómo acceder a los resultados en las sentencias de selección (SELECT).

 

Estos resultados quedan modelados por medio de la interfaz ResultSet. Como ya mencionamos anteriormente, la forma de obtener un objeto del tipo ResultSet es por medio de una llamada a executeQuery() (o execute() si es un procedimiento almacenado).

 

Una vez se ha devuelto el ResultSet, podemos avanzar por las distintas filas que lo componen empleando el método next(). Es imprescindible llamarlo al menos una vez, pues es este el que se encarga de desplazar el cursor interno de la base de datos.

 El método next() devuelve un booleano indicando si hay o no más filas. Por tanto el código típico de consulta de resultados es un bucle while. En el siguiente ejemplo, accedemos a una consulta generada sobre la tabla datos que ya presentamos anteriormente:

 

ResultSet rset = stmt.executeQuery (select * from datos);

while (rset.next()) {

         System.out.println (DNI:  + rset.getInt());

         System.out.println (Nombre:  + rset.getString());

}

 

Nótese que la forma de obtener los resultados es análoga a como los obteníamos de un procedimiento almacenado.

 

3.7.   Transacciones.

 

En algunas ocasiones es de vital importancia que varias actualizaciones sobre distintos registros de la base de datos se lleven a cabo de forma atómica. Para asegurarlo empleamos transacciones.

 

Un ejemplo típico de transacción es una transferencia entre dos cuentas bancarias. Si queremos transferir una cantidad de dinero de una cuenta a otra, tanto la retirada de la cantidad a transferir de la cuenta origen como la actualización de la cuenta receptora deben realizarse dentro de la misma transacción: o se ejecutan correctamente las dos operaciones o ninguna de ellas lo hace.

 

Las conexiones que obtenemos de la clase DriverManager están configuradas de tal forma que tras la ejecución de una sentencia se hace un commit, es decir, los cambios en la base de datos se hacen permanentes. Podemos cambiar esto, de tal forma que se ejecuten todas las sentencias SQL hasta que hagamos un commit explícito, por medio de una llamada al método setAutoCommit().

 

Así, para ejecutar varias sentencias dentro de la misma transacción, podemos emplear un fragmento de código como el siguiente:

 

con.setAutoCommit (false);

stmt1.executeQuery();

stmt2.executeUpdate();

stmt3.execute();



con.commit(); // Hacemos permanentes todos los cambios de una sola vez.

 

En lugar de hacer permanentes los cambios, podríamos haberlos retraído todos, por medio de un rollback. En este caso, deberíamos haber invocado el método rollback() sobre el objeto de tipo Connection. Esto mismo debería hacerse en el caso en que se lance una excepción mientras se procesa una excepción.

 

Puede cambiarse también el tipo de transacción a realizar (siempre que no nos encontremos en la mitad de una transacción).  Esto se consigue por medio del método de la interfaz Connection setTransactionIsolation().

Este método toma un parámetro de tipo entero, existiendo sendas constantes con los distintos tipos de transacciones. El método anterior intenta fijar el nivel de transacción al requerido, pero puede que la base de datos no lo soporte. En este último caso, podemos usar la llamada

con.getMetaData().supportsTransactionIsolationLevel (n_transacción)

 para preguntar si la base de datos soporta el nivel de transacción al que se quiere pasar.

 

El valor de las constantes indicando el nivel de transacción puede ser uno de los siguientes:

 

q       TRANSACTION_NONE. No se soportan las transacciones.

q       TRANSACTION_READ_COMMITTED. Se previenen las lecturas sucias, aunque pueden darse lecturas no repetibles y fantasma.

q       TRANSACTION_READ_UNCOMMITTED. Pueden darse lecturas sucias, lecturas no repetibles y fantasma.

q       TRANSACTION_REPEATABLE_READ. No pueden darse lecturas sucias ni lecturas no repetibles, aunque sí pueden darse lecturas fantasma.

q       TRANSACTION_SERIALIZABLE. Es el nivel más restrictivo de todos. No pueden darse lecturas sucias, ni no repetibles ni fantasma.

 

4.   Estructura de un programa que usa JDBC.

 

Un tema que no hemos tratado hasta el momento es la gestión de las conexiones. Las conexiones, sentencias y conjuntos de resultados son recursos escasos.  Es por esto que cuando uno de estos objetos no se usa, debería liberarse, sin esperar al recolector de basura.  La forma más elegante de hacerlo es usando la claúsula finally de los bloques try-catch-finally.  Un esquema típico es el siguiente:

Connection con = null;

Statement stmt = null;

ResultSet rset = null;

try {

         << Usar con, stmt y rset >>

} catch (SQLException sqle) {

         

} catch () {

         

} finally {

         try {

                   if (rset != null) rset.close();

                   if (stmt != null) stmt.close();

                   if (con != null) con.close();

         } catch (Exception ignoraMe) {}

}

 

Nótese que los objetos se cierran en el orden inverso al que se adquirieron, pues en otro caso se lanzarían excepciones.

 

Para finalizar este tema, vamos a escribir un programa que contacte con una base de datos ODBC por medio del puente JDBC-ODBC.  Además, veremos como configurar el gestor de ODBC correctamente en una máquina basada en Windows.  Empezaremos ahora por este punto.

4.2.   Ejemplo.

 

Esta aplicación usa AWT para mostrar una interfaz gráfica desde la que ejecutar las sentencias SQL.

 

También se muestra en este ejemplo cómo tratar las excepciones SQLException, además del uso de la interfaz ResultSetMetaData, que permite obtener alguna información acerca de los conjuntos de resultados y que es especialmente útil para programas que no conocen en tiempo de compilación la longitud, tipos y características de los resultados.

 

import java.awt.*;

import java.awt.event.*;

import java.sql.*;

 

public class ConsolaJDBC extends Frame implements ActionListener

{

  Connection con = null;

 

  TextField tf;

  TextArea ta, taRes;

 

 

  public ConsolaJDBC () {

    super ("Consola JDBC");

 

    // Creamos una zona en la que representar y

    // modificar el DSN actual.

    Panel p = new Panel ();

    p.add (new Label ("DSN"));

    tf = new TextField (30);

    p.add (tf);

    Button bt = new Button ("Conectar");

    bt.addActionListener (this);

    p.add (bt);

 

    setLayout (new BorderLayout ());

   

    Panel paux = new Panel();

   

    paux.add (p);

 

    // Creamos una zona en la que se encuentra la entrada

    // para la sentencia SQL.

    p = new Panel ();

    p.add (new Label ("Sentencia SQL"));

    ta = new TextArea (7, 30);

    bt = new Button ("Ejecutar");

    p.add (bt);

    bt.addActionListener (this);

 

    p.add (ta);

   

    paux.add (p);

 

    // Incluimos una TextArea en la que mostrar los resultados.

    taRes = new TextArea ();

 

    add (paux, BorderLayout.CENTER);

    add (taRes, BorderLayout.SOUTH);

 

    // Ponemos un font de ancho fijo.

    Font font = new Font ("Courier", Font.PLAIN, 12);

    ta.setFont (font);

    taRes.setFont (font);

    tf.setFont (font);

 

    setSize (500, 400);

    setVisible (true);

  }

 

  public void actionPerformed (ActionEvent ae) {

    String str = ae.getActionCommand();

   

    Statement stmt = null;

    ResultSet rset = null;

      

    if (str.trim().equals("")) return;

 

    try {

      if (str.equals ("Conectar")) {

         // Primera vez.

         if (con == null)

           // Cargamos el puente JDBC-ODBC que se encuentra en

           // la siguiente clase.

           Class.forName ("sun.jdbc.odbc.JdbcOdbcDriver");

         else

           con.close();

        

         con = DriverManager.getConnection ("jdbc:odbc:" +

                                                  tf.getText().trim(), "", "");

         // Dejamos el usuario y el password a blanco.

      }

      else {

         // Ejecución de una sentencia SQL.

         if (con == null) return;

 

         stmt = con.createStatement ();

        

         String stmtStr = ta.getText();

 

         taRes.append ("\n\n--> Sentencia: " + stmtStr);

        

         if (stmtStr.trim().toUpperCase().startsWith ("SELECT")) {

           rset = stmt.executeQuery (stmtStr);

 

           imprimeResultados (rset);

         }

         else

           taRes.append ("\nElementos actualizados: " +

                            stmt.executeUpdate (stmtStr));   

      }

 

      taRes.append ("\nOK");

    } catch (SQLException sqle) {

      trataSQLException (sqle);

    } catch (ClassNotFoundException cnfe) {

      taRes.append ("\n--- Excepción:\n");

      taRes.append ("===================\n");

      taRes.append (cnfe.toString());

    } finally {

      try {

         if (rset != null) rset.close();

         if (stmt != null) stmt.close();

         // No cerramos la conexión, podemos seguir

         // usándola.

         ta.setText ("");

 

      } catch (Exception e) {}

    }

  }

 

 

  private void imprimeResultados (ResultSet rset) throws SQLException {

    // Obtenemos algunos datos acerca del resultado.

    ResultSetMetaData rsmd = rset.getMetaData();

   

    int ncols = rsmd.getColumnCount ();

  

    taRes.append ("\n\n");

    for (int i = 1; i <= ncols; i++)

      taRes.append (rellena (rsmd.getColumnLabel (i)) + "  ");

   

    taRes.append ("\n==============================================\n");

   

    while (rset.next()) {

      for (int i = 1; i <= ncols; i++) {

         // Nótese que siempre cogemos los resultados

         // como cadenas.  El método hace la conversión

         // por nosotros siempre que sea posible.

         String strRes = rset.getString(i);

        

         taRes.append (rellena (strRes) + "  ");

      }

      taRes.append ("\n");

    }

  }

 

  private String rellena (String strRes) {

    // Rellenamos con espacios hasta 10.

    for (int j = strRes.length(); j < 15; j++)

      strRes  = " " + strRes;

        

    return strRes;

  }

 

  private void trataSQLException (SQLException sqle) {

    taRes.append ("\n--- Excepción:\n");

    taRes.append ("===================\n");

   

    while (sqle != null) {

      taRes.append (sqle.toString());

 

      sqle = sqle.getNextException();

      // Continuamos con la siguiente excepción encadenada.

    }

  }

 

  public static void main (String[] args) {

    new ConsolaJDBC();

  }

}

5.   Optimizaciones y otras mejoras.

Vamos a ver, a continuación, algunas mejoras y optimizaciones que podemos incluir en nuestras aplicaciones.

Tienen que ver con eficiencia, la actualización bases de datos mediante un ResultSet y con el manejo de los metadatos de una base de datos.

Recordemos que ya hemos estudiado una primera optimización: el uso de sentencias preparadas.

Dichas sentencias mejoran la eficiencia, sólo se compilan una vez, así como la reutilización de sentencias SQL mediante el uso de parámetros.

5.1.   Porcentaje de carga.

La mayoría de los drivers JDBC se encuentran optimizados or defecto para suministrar las máximas prestaciones en relación con una base de datos concreta. No obstante, tenemos métodos que nos permiten refinar el comportamiento de un ResultSet para determinada aplicación

Un ResultSet aparece como resultado de una consulta a la base de datos. Como es sabido, el número de filas conformes a dicha consulta puede ser cualquiera, desde ninguna a millones.

Por tanto, un ResultSet no puede comprometerse a almacenar todas las filas de una consulta en la memoria del programa cliente.

Más bien guarda unas cuantas, y solicita más a la base de datos a medida que se necesitan. El número de filas que se almacena en local viene definido por el porcentaje de carga (fetch size).

El porcentaje de carga puede modificarse a nivel de ResultSet y/o a nivel de Statement, mediante el método setFecthSize.

Si lo modificamos a nivel se sentencia, todos los ResultSet obtenidos bajo esa sentencia lo adquieren.

Un Resultset puede modificar en cualquier momento su porcentaje de carga.

Algo similar ocurre con la dirección en la que se procesan las filas. Puede ser controlada mediante el método setFetchDirection, tanto a nivel de Statement como de ResultSet.

Hay que tener cuidado con estas optimizaciones, ya que alterar los valores por defecto puede conseguir el efecto contrario al deseado.

5.2.   Actualizaciones en modo batch 

Cada vez que enviamos una orden a la base de datos, JDBC la envuelve en una transacción y hace, si  no hay errores, un commit automático, incluso con sentencias preparadas.

Por otra parte, el envío en sí de una orden SQL a un servidor remoto de base de datos significa aumentar l tráfico de la red, amén del tiempo que se consume en dicho envío.

Las actualizaciones en modo batch permiten enviar varias órdenes a la base de datos dentro de la misma transacción y de una sola vez.

Lo primero que hemos de hacer es tomar nosotros el control de las transacciones.

En segundo lugar, añadimos sentencias SQL mediante el método addBatch.

Por último, las ejecutamos mediante executeBatch.

Las órdenes SQL sólo pueden devolver el número de filas afectadas, esto es, no podemos usar consultas.

El gestor de bases de datos ejecuta el paquete de órdenes recibidas en secuencia, y devuelve el número de filas afectadas por cada una. Esta información llega a nosotros en forma de una array de enteros devuelto por executeBatch.

Si alguna de las órdenes produce un error, se lanzará una BatchUpdateException.

Por ejemplo:

Statement st = con.createStatement();

con.setAutoCommit(false);

st.addBatch(INSERT INTO.);

st.addBatch(INSERT INTO.);

st.addBatch(INSERT INTO.);

int resultados[] = st.executeBatch();

.

Es possible limpiar la lista de órdenes SQL mediante el método clearBatch.

Las actualizaciones en modo batch pueden realizarse con sentencias preparadas y con procedimientos almacenados.

No todos los drivers soportan actualizaciones en modo batch. Como de costumbre, podemos recurrir a DatabaseMetaData para investigar en tiempo de ejecución si nuestro driver lo permite (método supportBatchUpdates).

5.3.   Nuevos tipos de ResultSet 

Los ResultSet que hemos estudiado hasta ahora se caracterizan porque podemos recorrerlos en un solo sentido, de la primera fila hasta la última, no muestran actualizaciones de sus filas si otro proceso las ha modificado, y son de sólo lectura, esto es, no podemos modificar la base de datos desde su interior.

JDBC 2.0 permite la creación de ResultSets con nuevas funcionalidades, pero antes de introducirnos en ellos, es preciso no olvidar dos puntos:

1.    A mayor funcionalidad, menor rendimiento.

2.    Lo que estos tipos de ResultSet nos proporcionan, en general podemos realizarlo sin recurrir a ellos.

Las nuevas características de los ResultSets podemos clasificarlas en tres apartados:

Movimiento a través del ResultSet: en terminología de Sun, scrollable resultset.

El ResultSet es sensible a los cambios en sus filas realizados por otros procesos: scrollable sensitive resultset.

El ResultSet puede emplearse para modificar la base de datos:concurrently updatable resultset.

Dentro de la interfaz ResultSet, tenemos constantes que definen lo anterior. Por ejemplo:

final int sensibilidad = ResultSet.TYPE_SCROLL_SENSITIVE;

final int sensibilidad = ResultSet.CONCUR_UPDATABLE;

Statement st = con.createStatement(sensibilidad,concurrencia)

Acabamos de crear un ResultSet a través del cual podemos:

movernos en cualquier dirección, ver cambios en sus filas producidos por otros procesos y actualizar la base da datos.

Para obtener más tipos, basta con combinar los posibles valores de las constantes.

Téngase en cuenta que no todos los drivers soportarán estos tipos de ResultSet aunque aseguren ser conformes a JDBC 2.0.

5.4.   Manejo de metadatos.

Un metadato representa,información que describe la estructura y propiedades de los datos y no de sus contenidos.

JDBC nos proporciona meta-información acerca de un ResultSet y de la base de datos con la que estamos conectados, de manera que podamos consultarla en tiempo de ejecución y tomar decisiones en consecuencia.

Para un ResultSet, tenemos la interfaz ResultSetMetaData. Por ejemplo:

ResultSet rs = st.executeQuery(SELECT * FROM CONTACTOS)

ResultSetMetaData  rsmd = rs.getMetaData();

String nombre = rsmd.getColumnName(indice);

String tipo = rsmd.getColumnTypeName(indice);

int nfilas = rsmd.getColumnCount();



La interfaz DatabaseMetaData nos permite averiguar en tiempo de ejecución las características de la base de datos que estamos manejando.

Dada la cantidad de métodos que poseen ambas interfaces, se discutirán los más importantes sobre la documentación de las APIs de JDBC.

Empleando estas interfaces, al precio de complicar el programa, podemos asegurarnos de que las características JDBC requeridas están disponibles bajo el driver que estemos usando en cada momento.

 

6.     Nuevos tipos de datos SQL3.

 JDBC 2.0 nos da cobertura para los nuevos tipos de datos SQL3, que forman parte del ANSI SQL. Cada uno de ellos se representa en Java, como de costumbre, mediante una clase o una interfaz.

Recuérdese que no todos los drivers (a día de hoy) los soportarán. También ocurre a menudo que una base de datos concreta posee soporte para todos o algunos de estos tipos, pero empleando su propia sintaxis.

Nosotros los abordaremos desde la especificación ANSI del lenguaje SQL.

6.1.        Tipos simples definidos por el usuario (Distinct).

Mediante esta funcionalidad, podemos crear tipos de datos basados en otros nativos de SQL.

Más que crear nuevos tipos de datos, lo que estamos haciendo es definir sinónimos.

Consideremos, por ejemplo, una aplicación que trabaje con el código postal. Podríamos escribir lo siguiente:

CREATE TYPE CODPOSTAL AS CHAR(5).

Acabamos de definir un sinónimo para CHAR(5), el cual podemos utilizar para especificar el tipo de datos de una columna o de un tipo estructurado.

Algunas bases de datos requieren una sintaxis ligeramente distinta:

CREATE DISTINCT TYPE CODPOSTAL AS CHAR(5).

Puesto que estos tipos de datos se refieren a otros nativos de SQL, no existen para ellos clases ó interfaces específicas en el lado Java, sino que se manejan en función del tipo SQL.

En nuestro ejemplo, CHAR(5) para JDBC es una String, por lo que para leer y escribir un campo de tipo CODPOSTAL usaríamos los métodos getString y updateString, respectivamente.

6.2.        Objetos binarios de gran tamaño (Blob).

En las aplicaciones actuales, cada vez es más común la necesidad de manejar información no estructurada, es decir, tiras de bytes cuyo significado sólo es conocido por las aplicaciones, no por la base de datos.

Dicha información suele caracterizarse, además, por ser muy voluminosa. Un ejemplo típico podría ser una aplicación que manejara información multimedia, como música y vídeo.

El tipo de datos BLOB (Binary Large Object) es la respuesta de SQL a esta necesidad de manipular datos no estructurados.

En el lado Java, tenemos la interfaz Blob para manipularlos. Las interfaces ResultSet, PreparedStatement y CallableStatement poseen métodos de tipo getBlob y updateBlob.

Por ejemplo:

final String ordenSQL = SELECT VIDEO FROM PELICULAS WHERE PELICULA_ID BETWEEN 1 AND 7;

Statement st = con.createStatement();

En este código suponemos que la columna VIDEO es de tipo Blob.

Supongamos que queremos guardar una copia de estas películas en el sistema de archivos. Una posible forma de hacerlo sería:

ResultSet rs = st.executeQuery(ordenSQL);

While(rs.next()){

Blob b = rs.getBlob(1);

InputStream is = b.getBinaryStream();

Resto del código para grabar el archivo...

}

Un BLOB puede ser muy grande, hablamos de Gb en el caso de una película, por ejemplo, y no es factible almacenarlo directamente en la memoria del programa cliente.

JDBC resuelve el problema empleando, tras el escenario, un SQL Locator, es decir, un puntero. El programa cliente, al invocar a getBlob no obtiene los datos, sino una referencia a donde se encuentran.

Para acceder a ellos, es necesario materializar el Blob, según la terminología JDBC. Existen varias formas para hacer esto. Una de ellas es la que se muestra en el ejemplo.

6.3.        Objetos carácter de gran tamaño (Clob).

El tipo de datos CLOB (Carácter Large Object) representa la misma cosa y responde a idéntica necesidad que un BLOB, sólo que el tipo de datos es texto, caracteres.

La interfaz correspondiente se llama Clob, y se le pueden aplicar las mismas consideraciones de propósito y uso que en el epígrafe anterior.

Por ejemplo:

final String ordenSQL = SELECT TEXTO FROM BOE WHERE BOE_ID = 3;

Statement st = con.createStatement();

ResultSet rs = st.executeQuery(ordenSQL);

While(rs.next()){

Clob c = rs.getClob(1);

Reader r = c.getCharacterStream();

Resto del código para grabar el archivo...

}

6.4.        Matrices (Array).

Un ARRAY representa una colección consecutiva de datos. Java modela esto mediante la interfaz Array.

Podemos aplicar a los arrays las mismas consideraciones de uso y propósito que comentamos para Blobs y Clobs.

No obstante, para materializar un Array, podemos seguir dos caminos: como un array Java del tipo de dato correspondiente o como un ResultSet.

Por ejemplo:

final String ordenSQL = "SELECT PROVINCIAS FROM PAISES WHERE PAIS_ID = ES";

ResultSet rs = stmt.executeQuery(ordenSQL);

while (rs.next()) {

Array p = rs.getArray("PROVINCIAS");

String[] provincias = (String[])p.getArray();

for (int i = 0; i < provincias.length; i++) {

// Código que hace uso de las provincias

}

}

Obsérvese que primero obtenemos el array y luego lo materializamos, en este ejemplo como un array Java del tipo de datos conveniente.

6.5.        Tipos de datos estructurados (Struct).

El tipo STRUCT permite la creación de nuevos tipos de datos más que la definición de sinónimos para los ya existentes.

Java modela esta noción mediante la interfaz Struct.

Veamos un ejemplo de la creación de un tipo de datos estructurado en ANSI SQL:

CREATE TYPE PERSONA{

NOMBRE VARCHAR(40),

APELLIDO VARCHAR(80),

TELEFONO VARCHAR(10)

}

Un tipo estructurado puede contener sinónimos. Si, por ejemplo, decidimos definir el número de teléfono como:

CREATE TYPE NUMERO_TELEFONO AS CHAR(10)

Entonces podríamos escribir:

CREATE TYPE PERSONA{

NOMBRE VARCHAR(40),

APELLIDO VARCHAR(80),

TELEFONO NUMERO_TELEFONO

}

El tipo de datos PERSONA puede ahora usarse como tipo para una columna o como atributo de otro tipo estructurado.

En el lado Java, para manipular Struct empleamos los métodos getObject, updateObject y setObject.

Un ejemplo de uso podría ser el siguiente:

final String ordenSQL = "SELECT INDIVIDUO FROM AGENDA WHERE INDIVIDUO _ID = 1";

ResultSet rs = stmt.executeQuery(ordenSQL);

rs.next();

Struct resultado = (Struct) rs.getObject(1);

Obsérvese como usamos getObject para acceder a la estructura. Ahora hemos de obtener los valores almacenados en su interior.

Object[] valores = resultado.getAttributes();

for( int i = 0; i < valores.length(); i++)

System.out.println(valores[i]);

Mediante el método getAttributes.

6.6.        Referencia a un tipo de datos estructurado(Ref).

El tipo de datos REF representa en SQL una referencia, un puntero, a un tipo de datos estructurado, y puede emplearse como columna de una tabla.

Una referencia normalmente se emplea para no duplicar información en la base de datos.

Supongamos que estamos modelando la noción de que en una empresa de franquicias, un jefe es responsable de potencialmente varios establecimientos.

Podríamos hacer lo siguiente:

CREATE TYPE JEFE{

JEFE_ID INTEGER,

NOMBRE VARCHAR(40),

APELLIDO VARCHAR(80),

TELEFONO VARCHAR(10)

}

Usando un tipo estructurado para definir qué entendemos por jefe.

Siguiendo la misma lógica:

CREATE TABLE ESTABLECIMIENTOS{

ID INTEGER,

DIRECCION VARCHAR(80),

RESPONSABLE JEFE

}

Si lo hacemos de esta forma, estamos duplicando información en cuanto un jefe sea responsable de más de un establecimiento. Es en situaciones de este tipo cuando podemos emplear referencias.

Así, escribiríamos:

CREATE TABLE JEFES OF JEFE (OID REF(JEFE) VALUES ARE SYSTEM GENERATED)

Definimos una tabla de referencias a un tipo estructurado. Cada vez que se hace una inserción en esta tabla, la base de datos crea una instancia de tipo REF(JEFE) y le asigna un identificador único en la columna OID.

Podemos ahora modificar adecuadamente la tabla ESTABLECIMIENTOS, de esta forma:

CREATE TABLE ESTABLECIMIENTOS{

ID INTEGER,

DIRECCION VARCHAR(80),

RESPONSABLE REF(JEFE)

}

De modo que un mismo jefe puede ser responsable de varios establecimientos, como queríamos, pero sin duplicar información.

En el lado Java, las referencias se modelan mediante la interfaz Ref.

Un posible ejemplo de uso sería:

final String ordenSQl = SELECT OID FROM JEFES WHERE JEFE_ID = 1;

ResultSet rs = st.executeQuery(ordenSQL);

rs.next();

Ref jefe = rs.getRef(OID);

Con lo que accederíamos a los datos del jefe cuyo identificador fuera 1, de esta forma:

Struct resultado = (Struct) jefe.getObject();

Object[] valores = resultado.getAttributes();

Tratando a la referencia como un puntero al tipo de datos estructurado JEFE.

Por otra parte, para insertar un responsable en un establecimiento, la sentencia SQL debería ser la siguiente:

INSERT INTO ESTABLECIMIENTOS

VALUES(

100,

GRAN VÍA, 40,

 SELECT OID FROM JEFES WHERE JEFE_ID = 1

)

Que realizaríamos desde el lado Java mediante las sentencias habituales.
